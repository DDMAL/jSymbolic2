jSymbolic Version History

All updates stated below are since jSymbolic 1.0

Currently Being Resolved:

  • There is a known bug related to saving features as Weka ARFF or CSV files when windowed feature extraction is being used. Works when saving ACE XML windowed features, however, and Weka ARFF and CSV files can be correctly saved when full-song processing is used rather than windowed processing.

jSymbolic 2.0:

  • Added the ability to extract features from MEI files using our new custom-built jMei2Midi project (which can convert MEI files to MIDI), and which in turn uses the jmei library. Of course, as MEI itself is still being developed, new aspects of the MEI format incorporated after the publication of this code may not yet be implemented.
  • Added the ability to implement and extract specialized MEI-specific features (i.e. features that cannot be extracted from MIDI files). An MEIFeatureExtractor architecture similar to the MIDIFeatureExtractor has been added in order to accomplish this. More information can be found at Extending Software page.
  • The new R-36 Number of Grace Notes feature was implemented as an example of an MEI-specific feature.
  • The ability to extract features separately and individually from multiple windows in a piece was added (as opposed to only being able to extract features from pieces overall, which was previously the only option). The ability to use windows with or without offsets was included.
  • Extracted features may now be saved as basic CSV text files or as Weka ARFF files (previously ACE XML was the only output option).
  • Added a new configuration file format to facilitate saving and reuse of settings, as well as to facilitate batch processing still further. This configuration file can be used the GUI, command line interface and API. It can be edited directly with a text editor, or may be saved from the GUI. The configuration file is also designed to be easily extensible.
  • Added a relatively detailed HTML manual.
  • Implemented an easy-to-use API so that jSymbolic can be incorporated into external software without needing to resort to command line calls.
  • Improved the GUI design in order to make it easier to use.
  • Improved the command line interface implementation in order to enable access to more functionality and to make it more easily extensible.
  • Set up jSymbolic to run as a separate process to deal with potential issues associated with the Java heap size. The user thus no longer has to worry about JVM-specific command line switches.
  • Improved and streamlined error handling. Among other things, now logs files that cannot be processed during batch processing, and does not terminate execution unnecessarily when problems occur.
  • Added substantial unit testing.
  • Updated support libraries (ACE and UtilityClasses).

jSymbolic 1.2.1:

  • Added a command line interface to complement the already-implemented GUI. This involved updating the FeatureSelectionPanel class with a new public getAllAvailableFeatureExtractors method to provide access to the available features, as well as updating the private populateFeatureExtractors method to now use this new method to set up features for the GUI internally. New features that were previously added using the populateFeatureExtractors method should now be added with the getAllAvailableFeatureExtractors method.
  • Updated support libraries (ACE and UtilityClasses).

jSymbolic 1.2:

  • Updated support libraries (ACE and UtilityClasses).

jSymbolic 1.1:

  • Imported jSymbolic into the NetBeans framework to assist further development.
