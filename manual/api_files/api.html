Using the API

EXPLANATION OF THIS SECTION

This section describes how jSymbolic's functionality may be accesed programmatically using its API.

OVERVIEW OF JSYMBOLIC'S API

jSymbolic's API (Application Programming Interface) allows programmers to carry out jSymbolic feature extractions directly from their own code, without needing to resort to using jSymbolic's via its command line interface (which, of course, is also an option). This can be done by calling certain public methods in the jSymbolic classes, as specified below, which have been specifically designed for this purpose. jSymbolic's API can be accessed by other Java code, or using other platforms that include functionality for accessing Java APIs, such as Matlab.

USING JSYMBOLIC'S API

In order to use jSymbolic's API, a programmer need simply place the jSymbolic.jar file (and its associated lib folder) in the same directory, and then point their code to it (and potentially adding it to their variables such as CLASSPATH or PATH). It should be noted that Java 8 or higher should be used, as jSymbolic does use some functionality that was only implemented beginning in this version of Java.

It is recommended that programmers download the development version of jSymbolic rather than the user version. Although the API is accessible from either distribution, the developer version has the substantial advantage of providing Javadoc documentation, as well as the well-commented jSymbolic source code itself.

Potential users of the API are encouraged to first read the section of this manual on the command line interface, as this will familiarize them with how jSymbolic processing works in general. They may also wish to read the section of this manual on jSymbolic's configruation settings file format, if they wish to use this type of file in their processing (it is optional).

In order to use the API, users should look at the Javadoc documentation or source code for the two classes in the jsymbolic.api package.

The first thing to do is to instantiate a JsymbolicProcessor object using the desired constructor, at which point the user in effect supplies the extraction settings to use with all feature extractions that will be carried out by the new object. Which constructor is used depends on whether one wishes to provide extraction settings directly as arguments or using a configruation settings file, and on whether one wishes to manually specify features to be extracted or simply wishes to extract the default features. Feature extraction can then be carried by calling one of this object's extract . . . methods, depending on how one wishes to specify the files to extract features from, and on whether one simply wishes to have extraction results saved to disk, or would prefer to also have results returned in the form of a JsymbolicData object. Note that some of the data stored in JsymbolicData objects is stored as objects belonging to classes from the related jMIR ACE and jMei2Midi projects. Output files are also saved, depending on what was passed to the JsymbolicProcessor constructor.

ADDITIONAL DETAILS ON JSYMBOLIC'S API

  • Instantiating a JsymbolicProcessor object by specifying extraction parameters as constructor arguments

    • The requires constructor requires that the following be specified as arguments: whether to save features separately for analysis windows, whether to only save features for music files as a whole, the analysis window size (if any), the analysis window overlap (if any), a list of feature names to extract, the save path for the ACE XML Feature Values file to be generated, the save path for the ACE XML Feature Definitions file to be generated, and whether to save extraced features in ARFF and/or CSV files as well as ACE XML files. There is one variant of this constructor that specifies the features to be extracted manually, and another which simply causes the default features to be extracted. Either windowed or overall features should be set to be extracted, but not both. If only overall file feature extraction is specified, then the specified window size and window overlap will have no effect on the processing. Each specified feature name must be a valid jSymbolic feature name, otherwise an exception will be thrown during instantiation.

  • Instantiating a JsymbolicProcessor object by specifying a configuration settings file as a constructor argument

    • The specified configuration settings file must exist and must contain all required information and be properly formatted. It is important to note that any input files specified in the configuration settings file will simply be ignored (during instantiation) if the jSymbolic API is being used.

  • Using the JsymbolicProcessor .extractAndSaveFeaturesFromFileOrDirectory methods to extract features

    • There are two versions of this method: one will only extract features from a single specified file (or the contents of a single specified directory), and the other will extract features from a list of specified files and/or directories. Either way, extraction is carried out on the specified file(s) using the extraction settings specified at instantiation, and results are saved at the specified file paths.

  • Using the JsymbolicProcessor .extractAndSaveFeaturesFromConfigFile method to extract features

    • This method parses the files and/or directories to extract features from from a configuration settings file. Note that extraction is carried out on the specified file(s) using the extraction settings specified at instantiation, not any contrasting settings specified in the configuration file, and results are saved at the specified file paths.

  • Using the JsymbolicProcessor .extractReturnAndSaveFeaturesFromFile method to extract features from a directory containing music files

  • Using the JsymbolicProcessor .extractAndReturnSavedFeaturesFromDirectory method to extract features from a directory containing music files

    • This method will return a Map with the keys of input File objects and values of corresponding JsymbolicData objects resulting from feature extraction. They specific File keys can be obtained using the Map.keySet method. Input files that are not valid are skipped and not processed. Such files are noted in the list of String objects passed to this method, and an indication is also printed to standard out. Output files are also saved as specified during JsymbolicProcessor instantiation.
