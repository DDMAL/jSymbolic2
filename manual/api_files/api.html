Using the API

EXPLANATION OF THIS SECTION

This section describes how jSymbolic's functionality may be accesed programmatically using its API.

OVERVIEW OF JSYMBOLIC'S API

jSymbolic's API (Application Programming Interface) allows programmers to carry out jSymbolic feature extractions directly from their own code, without needing to resort to using jSymbolic's via its command line interface (which, of course, is also an option). This can be done by calling certain public methods in the jSymbolic classes, as specified below, which have been specifically designed for this purpose. jSymbolic's API can be accessed by other Java code, or using other platforms that include functionality for accessing Java APIs, such as Matlab.

USING JSYMBOLIC'S API

In order to use jSymbolic's API, a programmer need simply place the jSymbolic.jar file (and its associated lib folder) in the same directory, and then point their code to it (and potentially adding it to their variables such as CLASSPATH or PATH). It should be noted that Java 8 or higher should be used, as jSymbolic does use some functionality that was only implemented beginning in this version of Java.

It is recommended that programmers download the development version of jSymbolic rather than the user version. Although the API is accessible from either distribution, the developer version has the substantial advantage of providing Javadoc documentation, as well as the well-commented jSymbolic source code itself.

Potential users of the API are encouraged to first read the section of this manual on the command line interface, as this will familiarize them with how jSymbolic processing works in general. They may also wish to read the section of this manual on jSymbolic's configruation settings file format, if they wish to use this type of file in their processing (it is optional).

In order to use the API, users should look at the Javadoc documentation or source code for the two classes in the jsymbolic.api package.

The first thing to do is to instantiate a jSymbolicProcessor object using the desired constructor. Which constructor is used depends on whether one wishes to provide extraction settings directly as arguments or using a configruation settings file, and on whether one wishes to manually specify features to be extracted or simply wishes to extract the default features. Feature extraction can then be carried by calling either this object's computeJsymbolicData method or its computeJsymbolicDataDirectory method, depending on whether one wishes to extract features from a single file at a time or from all appropriate files in a directory. Either way, the user will have a jSybmolicData object returned to them for each processed file, from which extracted features and other information can be accessed programatically. Note that some of this data is presented as objects belonging to classes from the related jMIR ACE and jMei2Midi code. Output files will also be generated, depending on what was passed to the jSymbolicProcessor constructor.

ADDITIONAL DETAILS ON JSYMBOLIC'S API

  • Instantiating a jSymbolicProcessor object by specifying extraction parameters as constructor arguments

    • The requires constructor requires that the following be specified as arguments: whether to save features separately for analysis windows, whether to only save features for music files as a whole, the window size, the window overlap, a list of feature names to extract, the save path for the ACE XML Feature Values File, the save path for the ACE XML Feature Definitions File and whether to save extraced features in ARFF and/or CSV files as well as ACE XML files. There is one variant of this constructor that specifies the features to be extracted manually, and another which simply causes the default features to be extracted. Either windowed or overall features should be set to be extracted, but not both. If only overall file feature extraction is specified, then the specified window size and window overlap will have no effect on the processing. Each specified feature name must be a valid jSymbolic feature name, otherwise an exception will be thrown during instantiation.

  • Instantiating a jSymbolicProcessor object by specifying a configuration settings file as a constructor argument

    • The specified configuration settings file must exist and must contain all required information and be properly formatted. It is important to note that any input files specified in the configuration settings file will simply be ignored if the jSymbolic API is being used.

  • Using the jSymbolicProcessor.computeJsymbolicData method to extract features from a single input file

    • A single jSymbolicData object will be returned, which will hold an ACE DataBoard object, a jMei2MIDI MEISpecificStorage object and File pointers to the two output ACE XML files. Input files that are not valid are skipped and not processed. Such files are noted in the list of String objects passed to this method, and an indication is also printed to standard out. Output files are also saved as specified during jSymbolicProcessor instantiation.

  • Using the jSymbolicProcessor.computeJsymbolicDataDirectory method to extract features from a directory containing music files

    • This method will return a Map with the keys of input File objects and values of corresponding jSymbolicData objects resulting from feature extraction. They specific File keys can be obtained using the Map.keySet method. Input files that are not valid are skipped and not processed. Such files are noted in the list of String objects passed to this method, and an indication is also printed to standard out. Output files are also saved as specified during jSymbolicProcessor instantiation.
